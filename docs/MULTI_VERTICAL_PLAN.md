# پلن کامل: CRM و نوبت‌دهی چندعمودی (Multi-Vertical SaaS)

> هدف: تبدیل پلتفرم فعلی از «فقط سالن زیبایی» به یک **CRM + نوبت‌دهی واحد** برای چند صنف، با قالب و ساختار اختصاصی هر صنف و **پلن‌های اشتراک یکسان**.

---

## ۱) چشم‌انداز محصول

یک هسته مشترک (رزرو، مشتری، پرسنل، پیامک، بیعانه، حسابداری، باشگاه مشتریان، سوپرادمین) + لایه‌ی **عمودی (Vertical)** که تعیین می‌کند:

| لایه | چه چیزی فرق می‌کند |
|------|---------------------|
| قالب بصری | رنگ، فونت، هیرو، لندینگ، آیکون‌ها |
| اصطلاحات | سالن/کلینیک/مطب، پرسنل/پزشک، خدمت/ویزیت |
| داده‌های اولیه | دسته‌ها و خدمات نمونه |
| ماژول‌های اختیاری | گالری/نمونه‌کار (سالن)، پرونده بیمار (پزشکی – فاز بعد) |
| جریان رزرو | چندخدمت موازی vs یک ویزیت، انتخاب پزشک اجباری |

**پلن‌ها (پرو / پلاس / حرفه‌ای) برای همهٔ اصناف یکی می‌مانند**؛ فقط متن و seed قالب فرق می‌کند.

### اصناف فاز ۱

| کد | عنوان | مخاطب |
|----|--------|--------|
| `beauty_salon` | سالن زیبایی | آرایشگاه، اسکین، ناخن |
| `dental_clinic` | کلینیک دندان‌پزشکی | مطب/کلینیک دندان |
| `medical_practice` | مطب پزشکی / تراپی | دکتر عمومی، متخصص، تراپیست، روانشناس |

---

## ۲) معماری پیشنهادی

```
┌─────────────────────────────────────────────────────────┐
│  سوپرادمین (l.xpaydar.ir)                               │
│  سایت‌ها + نوع کسب‌وکار + پکیج + سرنخ + اکانت فروش      │
└──────────────────────────┬──────────────────────────────┘
                           │ business_type
┌──────────────────────────▼──────────────────────────────┐
│  هسته مشترک (Core)                                      │
│  sites, users, appointments, services, staff, payments  │
│  sms, accounting, club, survey, PWA, Bale                 │
└──────────────────────────┬──────────────────────────────┘
                           │ VerticalRegistry
        ┌──────────────────┼──────────────────┐
        ▼                  ▼                  ▼
   beauty_salon      dental_clinic     medical_practice
   قالب + لیبل       قالب + لیبل        قالب + لیبل
   + seed خدمات      + seed خدمات       + seed خدمات
```

### اصل طراحی
- جدول‌ها را rename نکنیم (`salon_settings` بماند) تا مهاجرت سبک بماند.
- همه‌ی تفاوت‌ها از **`business_type` + رجیستری قالب** بیاید، نه از فورک کد.
- فیچر گیت پکیج‌ها مشترک بماند؛ فقط UI فیچرهایی مثل گالری/پورتفولیو برای پزشکی مخفی/کم‌رنگ شود.

## ۳) فازبندی اجرا

### فاز ۰ — Foundation (همین الان شروع می‌شود) ✅
- [x] ستون `business_type` روی `sites`
- [x] کلاس `VerticalRegistry` (لیبل‌ها، رنگ پیش‌فرض، seed خدمات/دسته، سکشن لندینگ)
- [x] `Migrator::seedDefaults($siteId, $type)`
- [x] ساخت سایت در سوپرادمین با انتخاب نوع کسب‌وکار
- [x] برگرداندن `business_type` و `labels` در API تنظیمات عمومی
- [ ] جایگزینی تدریجی کپی‌های سخت‌کد «سالن» در وب/ادمین با `labels`

### فاز ۱ — قالب‌های بصری (۱–۲ هفته)
- ۳ تم CSS/Tailwind (یا توکن‌های رنگ/فونت per vertical)
- Home لندینگ اختصاصی هر قالب (هیرو، بخش‌ها، CTA)
- آیکون و تصاویر پیش‌فرض per vertical
- صفحات رزرو با اصطلاحات درست (ویزیت / نوبت / خدمت)

### فاز ۲ — تفاوت ساختاری سبک (۲–۳ هفته)
| موضوع | سالن | دندان | مطب |
|-------|------|-------|-----|
| انتخاب پرسنل در رزرو | اختیاری | معمولاً اجباری (دکتر) | اجباری |
| چندخدمت در یک نوبت | بله | محدود / یک پلن درمان | معمولاً یک ویزیت |
| گالری / نمونه‌کار | فعال | اختیاری | معمولاً خاموش |
| فیلدهای مشتری | نام/موبایل/تولد | + کدملی اختیاری | + کدملی / تاریخ تولد |
| یادداشت داخلی | یادداشت سالن | شرح درمان کوتاه | شرح مراجعه |

پیاده‌سازی با `vertical_config` و فلگ‌های رزرو در `booking_rules_json` — بدون جداول جدا در این فاز.

### فاز ۳ — CRM واقعی فروش و پذیرش (۳–۴ هفته)
- قیف سرنخ → مشتری (از سوپرادمین تا پنل سایت)
- پرونده مشتری غنی‌تر (تگ، منبع، یادداشت‌های زمانی)
- یادآور نوبت / عدم مراجعه (no-show)
- گزارش تبدیل و عملکرد پرسنل/پزشک
- وظایف پیگیری برای پذیرش

### فاز ۴ — ماژول‌های عمودی پیشرفته (اختیاری / بعداً)
**دندان:** طرح درمان چندمرحله‌ای، تصویر رادیوگرافی در پرونده  
**مطب:** نوع ویزیت (حضوری/آنلاین)، مدت پیش‌فرض پزشک، نسخه‌ی ساده (متن)  
**سالن:** پکیج چندجلسه‌ای، عضویت باشگاه پیشرفته  

فقط وقتی هسته پایدار شد.

### فاز ۵ — مارکت و رشد
- دموی هر قالب روی ساب‌دامین
- ویزارد راه‌اندازی ۱۰ دقیقه‌ای
- مارکت‌پلیس قالب (تم‌های بیشتر)
- اپ پذیرش (کیوسک) و تبلت

## ۴) پلن‌های اشتراک (ثابت برای همهٔ اصناف)

| | پرو | پلاس | حرفه‌ای |
|--|-----|------|----------|
| ماهانه | ۲٬۹۹۰٬۰۰۰ | ۴٬۹۹۰٬۰۰۰ | ۶٬۹۹۰٬۰۰۰ |
| سالانه | ۱۲× − ۱۵٪ | ۱۲× − ۱۵٪ | ۱۲× − ۱۵٪ |

## ۵) مدل دادهٔ پیشنهادی (تدریجی)

### الان (فاز ۰)
```text
sites.business_type  VARCHAR  -- beauty_salon | dental_clinic | medical_practice
```

### فاز ۲
```text
salon_settings.vertical_config_json  -- فلگ‌های UI/رزرو
users.national_id (اختیاری، nullable)
appointments.visit_type (اختیاری)
```

### فاز ۴ (فقط در صورت نیاز)
جداول جدا مثل `patient_records` / `treatment_plans` — نه زودتر.

## ۶) UX و قالب‌ها (خلاصهٔ جهت بصری)

| قالب | حس | رنگ پایه | تاکید هیرو |
|------|-----|----------|------------|
| سالن | لوکس، گرم، مد | سرخابی / طلایی ملایم | زیبایی و تجربه |
| دندان | تمیز، اعتماد، کلینیکال | آبی دریایی / سفید | آرامش و تخصص |
| مطب | حرفه‌ای، آرام، پزشکی | سبز زیتونی / سرمه‌ای | مراقبت و نوبت آسان |

قوانین طراحی محصول (مثل قبل): یک ترکیب در viewport اول، برند قوی، بدون کارت‌های تزئینی در هیرو.

## ۷) ریسک‌ها و تصمیم‌ها

| ریسک | تصمیم |
|------|--------|

## ۸) معیار موفقیت

1. ساخت سایت از سوپرادمین با ۳ نوع و لندینگ/خدمات نمونهٔ درست  
2. مشتری روی هر قالب بتواند نوبت بگیرد بدون سردرگمی اصطلاحات  
3. پلن‌ها بدون تغییر قیمت برای هر سه صنف کار کند  
4. تیم فروش بتواند در سرنخ‌ها نوع کسب‌وکار را ثبت و دمو نشان دهد
